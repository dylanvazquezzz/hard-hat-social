```sql
-- 009_jobs_hired_select.sql
-- Allow hired contractor to view jobs they were hired for

create policy "jobs: hired contractor select own"
  on jobs for select
  using (
    exists (
      select 1 from contractors
      where user_id = auth.uid()
      and id = jobs.hired_contractor_id
    )
  );
```

Create `app/jobs/actions.ts` as a `'use server'` file with three exported functions following the pattern in `app/admin/actions.ts`:

**createJob(data):** Accepts `{ gc_contractor_id: string, title: string, description: string, trade: string, location_city: string | null, location_state: string | null }`. Calls `getSupabaseAdmin().from('jobs').insert({ ...data, status: 'open' })`. Throws on error. Calls `revalidatePath('/jobs')`.

**markHired(jobId, hiredContractorId):** Updates `status: 'hired'`, `hired_contractor_id`, `hired_at: new Date().toISOString()` where `id = jobId`. Throws on error. Calls `revalidatePath('/jobs')`.

**markCompleted(jobId, hiredContractorId):** Updates `status: 'completed'`, `completed_at: new Date().toISOString()` where `id = jobId`. Throws on error. Calls `revalidatePath('/jobs')` AND `revalidatePath(\`/contractors/${hiredContractorId}\`)` — both are required so the portfolio section reflects completion immediately.

All three functions use `getSupabaseAdmin()` (not the browser client). Do not add any auth guard logic inside the actions — RLS and the DB trigger enforce correctness at the database layer.
